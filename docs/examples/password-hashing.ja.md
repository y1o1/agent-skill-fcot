# パスワードハッシュ化

> **主題:** 「パスワードは平文で保存せず、ハッシュ化すべきだ。」

## FCoT なし

```
> パスワードは平文で保存せず、ハッシュ化すべきだ。

まったくその通りです — これはアプリケーションセキュリティにおける最も根本的な原則の一つです。

ハッシュ化が不可欠な理由: データベースが侵害された場合（リソースが潤沢な組織でも侵害は起きます）、
平文パスワードが存在すると、攻撃者はすべてのユーザーアカウントに即座にアクセスできてしまいます。
さらに、人はパスワードを使い回すため、被害はあなたのシステムをはるかに超えて広がります。

ハッシュ化の仕組み: 暗号ハッシュ関数はパスワードを固定長の文字列に変換します。この変換は不可逆です。
ユーザーがログインする際、入力内容をハッシュ化して保存済みのハッシュと照合します — 元のパスワードを
保存したり再導出したりする必要は一切ありません。

適切なアルゴリズムを選ぶ: ハッシュはどれも同じではありません。MD5 や SHA-1 は高速ですが、
それ自体がリスクです。パスワード用に設計されたアルゴリズムを使用してください:
- bcrypt — 広くサポートされており、長年の実績がある
- Argon2 — Password Hashing Competition 優勝者、現在のベストプラクティス
- scrypt — メモリハード設計で、強力な代替選択肢

これらのアルゴリズムは意図的に低速であり、レインボーテーブル攻撃を防ぐための
ソルトが組み込まれています。

自前実装は禁物: 言語やフレームワーク向けに検証済みのライブラリを使用してください —
Node.js の bcrypt、Python の Werkzeug/passlib、Java の spring-security-crypto など。

この主張は正しく、アドバイスは広く確立されています。パスワードのハッシュ化は
現代のソフトウェアにおいて譲れない要件です。
```

## FCoT あり

```
> /fcot

## FCoT: パスワードは低速かつソルト付きのアルゴリズムでハッシュ化しなければならない（平文保存は不可）

反論 #2 は棄却されない。特定のレガシープロトコル（CHAP、NTLM）は、アーキテクチャ上
元の認証情報へのアクセスを必要とする。より正確な表現は次の通り: 「標準的なアプリケーション認証では、
パスワードはハッシュ化しなければならない...」
```

### 考察

#### 反論のテーブル

| # | 反論 | 棄却条件 | 検証 | 結果 |
|---|------|----------|------|------|
| 1 | 弱いアルゴリズム（MD5、ソルトなし SHA-1）によるハッシュ化は解読可能 — 無条件に「パスワードをハッシュ化せよ」と言うだけでは危険なほど不完全。 | 強力なアルゴリズムを明示的に推奨し、ソルト付きに言及していれば棄却。 | bcrypt、Argon2、scrypt を列挙し、「意図的に低速」でありソルトが組み込まれていると説明した。アルゴリズムを具体的に示したアドバイスだった。 | ✓ |
| 2 | 一部のレガシー認証プロトコル（CHAP、NTLM、PAP）では、一方向ハッシュ化が妨げるような形で、サーバー側が元のパスワードを再現または検証する必要がある。 | 普遍的な主張としてではなく、現代的な汎用ウェブ認証にスコープを絞っていれば棄却。 | 標準的なログインフローに暗黙的にスコープは限定されている。しかし冒頭の主張は無条件であり、読者が普遍的に適用する可能性がある。レガシープロトコルの制約への言及は一切ない。 | ✗ |
| 3 | パスワードが通信中（TLS なし）、ログ、またはメモリダンプで露出した場合、アプリケーション層でのハッシュ化は意味をなさない。 | ストレージセキュリティに正確にスコープを絞っていれば棄却。 | 「データベースが侵害された場合」という記述で、ストレージについて明示的に言及している。完全なセキュリティチェックリストを主張しているわけではない。 | ✓ |
| 4 | 規制された環境では、HSM や暗号化ストレージをハッシュ化の代替として検討できる。 | 暗号化（可逆）とハッシュ化（不可逆）を混同している反論であれば棄却。 | 保存時の暗号化はパスワード認証のためのハッシュ化と等価ではない。暗号化キーが漏洩すれば全パスワードが露出する。パスワードストレージ用の HSM も内部ではハッシュ化を行う。 | ✓ |
| 5 | 「言語の検証済みライブラリを使え」という助言は、非推奨または保守不十分なライブラリに対応していない。 | 一般的なソフトウェア保守上の懸念であり、平文よりハッシュ化すべきという判断の欠陥ではなければ棄却。 | この反論は、ライブラリガイダンスの完全性を問うものであり、コアとなる判断を問うものではない。依存関係を最新に保つことは、別途存在する普遍的な関心事だ。 | ✓ |

#### まとめ

この例は **FCoT が必ずしも結論を覆すわけではない** ことを示している。AI の最初の回答は正しかった — パスワードのハッシュ化は確立されたセキュリティ原則であり、実践的なアドバイス（bcrypt/Argon2/scrypt）も健全だった。

それでも FCoT は価値を発揮した。レガシー認証プロトコル（CHAP、NTLM）というエッジケースを浮かび上がらせたのだ。これらはアーキテクチャ上、元の認証情報へのアクセスを必要とする。標準的なウェブ認証という文脈では判断は維持されるが、より正確に表現できるはずだ。

これは重要な特性を示している: FCoT は反論のためのツールではない。判断が健全であれば、FCoT はそれを確認する。問題が見つかった場合、それは本物の問題であって — 作られた反論ではない。
