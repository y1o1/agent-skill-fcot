# パスワードのハッシュ化

> **主張:** 「パスワードは平文で保存せず、ハッシュ化すべきだ。」

## FCoTなし

```
> パスワードは平文で保存せず、ハッシュ化すべきだ。

まったくその通り――これはアプリケーションセキュリティの中でも最も基本的な原則のひとつだ。

ハッシュ化が不可欠な理由: データベースが侵害された場合（リソースが豊富な組織でも侵害は起きる）、
平文パスワードはすべてのユーザーのアカウントへの即時アクセスを攻撃者に与えてしまう。
パスワードの使い回しは珍しくないため、被害はシステムの外にまで広がる。

ハッシュ化の仕組み: 暗号ハッシュ関数は、パスワードを元に戻せない固定長の文字列に変換する。
ログイン時には入力値をハッシュ化して保存済みのハッシュと比較するだけでよく、
元のパスワードを保存したり再導出したりする必要はない。

適切なアルゴリズムを使う: ハッシュならなんでもいいわけではない。MD5やSHA-1は処理が速い――
それ自体がリスクだ。パスワード向けに設計されたアルゴリズムを使うこと:
- bcrypt — 幅広くサポートされており、長年の実績がある
- Argon2 — Password Hashing Competitionの優勝アルゴリズム、現在のベストプラクティス
- scrypt — メモリハード設計の強力な代替案

これらは意図的に処理を遅くし、レインボーテーブル攻撃を防ぐソルトが組み込まれている。

自前実装はしない: 言語・フレームワークが提供する検証済みライブラリを使うこと――
Node.jsならbcrypt、PythonならWerkzeug/passlib、JavaならSpring Security Cryptoなど。

この主張は正しく、推奨内容も広く確立されている。パスワードのハッシュ化は
モダンなソフトウェアにおいて交渉の余地がない。
```

よく確立されたセキュリティ原則に対する想定通りの回答だ。

## FCoT: パスワードは平文で保存せず、低速かつソルト付きのアルゴリズムでハッシュ化しなければならない

### 結論

**要修正（スコープの限定）。**

反論 #2 は成立する。「パスワードは平文で保存せず、ハッシュ化すべきだ」という主張は大多数のユースケースにおいて正しい。しかし特定のレガシー認証プロトコル（CHAP、NTLM、一部のRADIAS構成）は元の認証情報へのアクセスを必要とし、一方向ハッシュ化とアーキテクチャ的に相容れない。

より正確な表現は: *「標準的なアプリケーション認証では、パスワードをハッシュ化しなければならない……」* か、あるいはエッジケースを明示的に認める形だ。実際のアドバイス（bcrypt/Argon2/scrypt推奨、MD5/SHA-1回避）は健全であり、そのまま有効だ。必要な修正は内容の実質ではなく、スコープの精度に関するものだ。
