# モノレポ移行

> **テーマ:** 「このプロジェクトはモノレポに移行すべきだ。」

## FCoT なし

```
> このプロジェクトはモノレポに移行すべきだ。

このプロジェクトを見た限り、その提案には異論があります — 少なくとも、もう少し背景を聞かないと判断できません。

モノレポが真価を発揮するのは、密な相互依存がある場合です。複数のパッケージがコードを共有し、
横断的な変更をアトミックに行う必要があり、CI/CDを一か所で統括したい — そういった状況です。
Turborepo、Nx、Bazel といったツールのオーバーヘッドは、パッケージ間の結合度がそれを正当化するときに初めて見合います。

このプロジェクト — FCoT スキルのリポジトリ — は単一の明確なスキルとドキュメントで構成されています。
スコープは絞られていて、構造もシンプルです。モノレポが解決すべきリポジトリ間の連携コストが
存在している兆候はありません。今移行するのは、まだ存在しない問題のためにインフラを整備することになります。

よくある誤りは「複数のリポジトリがある」ことと「モノレポが必要」を混同することです。
モノレポは特定の問題への解決策です。それは、リポジトリ間の結合が独立したバージョン管理やデプロイを
苦痛にさせている状況です。その結合が存在しないなら、モノレポは複雑さを増やすだけです。

この移行で解決しようとしている具体的な痛みは何ですか？
「今は特にないけど、将来的には」という答えなら — それは YAGNI の領域です。
移行を勧める前に、実際の結合度の変化を把握したいと思います。
```

AI は反論し、具体的な課題を尋ねます。回答は妥当ですが、意思決定を二項対立として捉えています。「今すでに痛みがあるか、そうでなければモノレポは不要か」という構図です。

## FCoT あり

```
> /fcot
```

## FCoT: このプロジェクトへのモノレポ移行は、リポジトリ間の具体的な結合の証拠なしには正当化されない

### 反論

| # | 反論 | 棄却条件 | 検証 | 結果 |
|---|------|----------|------|------|
| 1 | モノレポは業界標準だ — Google、Meta、Microsoft、Shopify といった大企業が採用しているのだから、デフォルトはモノレポ寄りであるべきだ。 | これらの組織の規模やツール要件が、小規模な単機能プロジェクトと構造的に異なるなら棄却。 | Google は Piper/Blaze、Meta は Buck を必要とする — ほとんどのチームには入手できないカスタム VCS インフラだ。これらの組織は 10万人超のエンジニアとフルタイムのインフラチームで運営されている。このアナロジーは成立しない。 | ✓ |
| 2 | 先手を打った移行の方がコストが低い — 痛みが顕在化してから移行すると、より多くの歴史とコードを統合しなければならなくなる。 | プロジェクトの成長軌跡が高いリポジトリ間結合へと明確に向かっていないなら棄却。 | 現状だけでは確認できない。このリポジトリがマルチパッケージの SDK へ成長するなら、早期移行は構造的にコストが低い。ロードマップが不明な状態では、「痛みを待て」は系統的に遅すぎる判断になりうる。 | ✗ |
| 3 | 統一された CI、共有 lint、一貫したツール設定はモノレポの方が楽だ — この恩恵はパッケージ間の結合とは無関係に存在する。 | ツール設定の複製コストが低い程度の小規模プロジェクトであれば棄却。 | 現在の規模（スキルファイル 1 本、インストールスクリプト 1 本、README 2 本）では、設定の複製コストはわずかだ。3パッケージ以上になるまで、この恩恵は移行のオーバーヘッドを上回らない。 | ✓ |
| 4 | 「痛みを待て」には生存者バイアスがある — 先手を打った移行で痛みを回避したチームはケーススタディに登場しない。 | 生存者バイアスが対称的である場合 — 失敗したモノレポ移行も同様に過少報告されているなら棄却。 | どちらの方向にも記録された失敗例がある。バイアスの議論は対称であり、どちらの方向にも有利に働かない。 | ✓ |
| 5 | ユーザーはプロジェクトの将来の方向性について情報を持っているかもしれない — 確認せずに提案を退けると、十分に根拠のある判断を否定することになる。 | 回答がフラットな却下ではなく、その文脈を明確に求めていれば棄却。 | 回答は「この移行で解決しようとしている具体的な痛みは何ですか？」と明示的に尋ねており、推薦を条件付きとして枠組みした。 | ✓ |

### 結論

**要修正。**

反論 #2 は成立します。「痛みを待て」という経験則はデフォルトとして正しいですが、診断として不十分です。より正確な問いは「今、痛みを感じているか？」ではなく、「このプロジェクトの結合度の変化軌跡はどこへ向かっているか？」です。

**修正後の推薦:** 決断の前に二つの問いに答えること — (1) 12ヶ月後にどのようなパッケージが存在しているか、(2) パッケージをまたぐアトミックな変更はどれくらいの頻度で必要になるか。その答えが、移行が時期尚早か手遅れかを決めます。
