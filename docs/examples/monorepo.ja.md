# モノレポへの移行

> **題材:** 「このプロジェクトはモノレポに移行すべきだ。」

## FCoT なし

```
> このプロジェクトはモノレポに移行すべきだ。

このプロジェクトを見た感じ、その提案には少し待ったをかけたい——少なくとも、
もう少し背景情報がないと判断できない。

モノレポが力を発揮するのは、密な相互依存がある場合だ。複数パッケージ間でコードを
共有していて、アトミックなクロスカット変更が頻繁に発生し、CI/CD も一箇所で
まとめて管理したい——そういった状況ならツール（Turborepo、Nx、Bazel）の導入コストも
正当化できる。パッケージ間の結合度がそれを正当化するほど高い場合に限った話だ。

このプロジェクト——FCoT スキルリポジトリ——は、ドキュメント付きの単一スキルとして
完結している。スコープは狭く、構造も整理されており、モノレポが解決しようとしている
クロスリポジトリ間の調整コストが存在している証拠はどこにもない。今移行するのは、
まだ存在しない問題のためにインフラを積み上げることになる。

よくある誤解は、「リポジトリが複数ある」と「モノレポが必要」を混同することだ。
モノレポはあくまで特定の問題——リポジトリ間の結合が独立したバージョニングや
デプロイを困難にしている状態——に対するソリューションだ。その結合が存在しないなら、
モノレポは複雑性を増やすだけだ。

この移行で解決しようとしている具体的な痛みは何か？「今は特にないが、将来的には
あるかもしれない」という答えなら、それは YAGNI の領域だ。移行を推奨するには、
実際の結合度の推移を把握したい。
```

## FCoT あり

```
> /fcot

### 反論のテーブル

**要修正。**

**修正後の推奨:** まず 2 つの問いに答えること——(1) 12 ヶ月後にどんなパッケージが
増えるか、(2) クロスパッケージのアトミック変更はどの程度の頻度で必要になるか。
```

### 考察

#### 反論のテーブル

| # | 反論 | 棄却条件 | 検証 | 結果 |
|---|------|----------|------|------|
| 1 | モノレポは業界標準だ——Google、Meta、Microsoft、Shopify など主要組織が採用しているのだから、デフォルトはモノレポ寄りであるべき。 | それらの組織の規模とツール要件が、小規模な単機能プロジェクトと構造的に異なる場合は棄却。 | Google は Piper/Blaze、Meta は Buck を使用——ほとんどのチームには入手できないカスタム VCS インフラだ。これらの組織はフルタイムのインフラチームを持つ 10 万人超のエンジニアで動いている。類推は成立しない。 | ✓ |
| 2 | 事前移行の方が事後移行よりコストが低い——痛みが顕在化してから移行するということは、より多くの履歴とコードを抱えた状態で統合作業に臨むことになる。 | プロジェクトの成長軌道が高いクロスリポジトリ結合に向かっていないことが明確な場合は棄却。 | 現状だけでは確認できない。このリポジトリがマルチパッケージ SDK に成長するなら、早期移行の方が構造的に安い。ロードマップを把握せずに「痛みが出るまで待つ」と言うのは、系統的に遅すぎる判断になりかねない。 | ✗ |
| 3 | CI の統一、共通 lint、一貫したツール管理はモノレポの方が楽だ——これらのメリットはクロスパッケージ結合とは無関係に存在する。 | プロジェクトが小さく、ツール設定の重複が低コストで済む場合は棄却。 | 現在の規模（スキルファイル 1 本、インストールスクリプト 1 本、README 2 本）では、ツール設定の重複は些細な問題だ。パッケージが 3 つ以上にならない限り、このメリットは移行コストを上回らない。 | ✓ |
| 4 | 「痛みが出るまで待つ」には生存者バイアスがある——事前移行によって痛みを回避したチームはケーススタディに登場しない。 | 失敗したモノレポ移行も存在するが過小報告されており、生存者バイアスが対称である場合は棄却。 | 両方向に失敗事例のドキュメントがある。このバイアスの議論は対称であり、どちらの方向も優位に立たない。 | ✓ |
| 5 | ユーザーはプロジェクトの将来方向性について情報を持っているかもしれない——そのコンテキストを確認せずに提案を退けるのは、熟慮された判断を否定することになりかねない。 | 回答が平坦な拒絶ではなく、そのコンテキストを明示的に求めている場合は棄却。 | 回答は「この移行で解決しようとしている具体的な痛みは何か？」と明示的に問いかけ、推奨を条件付きで提示した。 | ✓ |

#### まとめ

AI の最初の回答は「具体的な痛みは何か？」と問いかけた——それ自体は妥当な問いだが、意思決定を*現在*の問題を中心に枠組みしている。FCoT はここに死角を発見した。

**事前移行の方が事後移行より安い場合がある。** プロジェクトのロードマップに密結合した複数パッケージが含まれるなら、痛みが顕在化してから移行するのは、より多くのコードと履歴を抱えた状態で作業することを意味する。「痛みが出るまで待つ」という発見的手法はデフォルトとして正しいが、より良い診断の問いは*現在の痛み*ではなく、*結合度の将来軌道*についてであるべきだ。

FCoT によって推奨は「痛みを見せてくれ」から「2 つの先を見た問いに答えてくれ」へと洗練された——意思決定のための、より実用的なフレームワークだ。
