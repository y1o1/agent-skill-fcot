# コードレビュー

> **Subject:** 「コードレビューには価値があり、すべてのチームが実施すべきだ」

## 期待される挙動（事前定義）

- **Control:** 強く同意する。メリット（バグ検出、知識共有、コード品質）を列挙する。
- **FCoT の結果:** 確認 — これは広く合意された原則。FCoT は反論を見つけるはずだが、すべて棄却すべき。edge case（代替としてのpair programming、小規模チーム）が浮上するかもしれないが、棄却されるべき。
- **浮上すべき主要な反論（すべて棄却されるべき）:**
  1. Pair programming が代替になる → 代替案であり、原則の否定ではない
  2. ベロシティが低下する → 短期コスト vs 長期品質のトレードオフで正当化できる
  3. ソロ開発者には不可能 → edge case であり、原則を否定しない
  4. 形式的なレビューは無価値 → 実行品質の問題であり、レビュー自体の否定ではない

## FCoTなし

```
おおむね同意するが、「すべてのチーム」という部分には留保が必要だ。

コードレビューには実証された本物のメリットがある: 本番環境に到達する前にバグを捕捉し、
チーム全体に知識を広め、一貫性を強制し、設計上の判断に自然なチェックポイントを作る。
研究（例: Microsoftのコードレビュー実践に関する研究）は一貫して、レビューがテストだけでは
見逃すような欠陥 — 特にロジックエラーと設計上の問題 — を捕捉することを示している。

ただし、コードレビューの価値は無条件ではない:

- **チームサイズが1名**: ソロ開発者が自分のコードをレビューすることで、時間をおいて
  新鮮な目で見るという恩恵は多少あるが、コードレビューの主要な価値 — 第二の視点 —
  は失われる。公式な自己レビュープロセスで部分的に代替できるが、同じではない。
- **形式的承認の文化**: 実質的なフィードバックなしに一貫してLGTMされるレビューは、
  レビューなしよりも悪い — プロセスのオーバーヘッドとレイテンシを加えながら、偽の
  安心感を生む。文化が本物のレビューをサポートしていないなら、その形式は無駄だ。
- **速度が重要なコンテキスト**: インシデント対応、hotfix、時間制限のある競技
  （hackathon、game jam）では、レビューでブロックするコストが未レビューのコードを
  リリースするリスクを上回る可能性がある。事後レビューはレイテンシなしに知識共有の
  メリットを得られる。
- **専門知識の非対称性**: チームの一人がドメインの唯一の専門家である場合、非専門家に
  よるそのコードのレビューは、本物の問題を捕捉するよりもスタイルの指摘に終わる可能性が
  ある。レビューには依然として知識移転の価値があるが、欠陥検出のメリットは薄れる。

より正確な主張: 「コードレビューはソフトウェアチームが利用できる最も効果的な品質プラクティスの
一つであり、ほとんどのチームが採用すべきだ — しかしその価値は文化、チーム構成、コンテキストに
依存する。真摯な関与なしに強制されたレビューは逆効果だ。」
```

## FCoTあり

```
### FCoT: 「コードレビューは最も効果的な品質プラクティスの一つであり、ほとんどの
チームが採用すべきだ — しかしその価値は文化、チーム構成、コンテキストに依存する」

#### 反論
| # | 反論 | 棄却条件 | 検証 | 結果 |
|---|------|----------|------|------|
| 1 | controlの例外はすべてコードレビューを放棄する理由ではなく、修正可能な問題だ。形式的承認? 文化を変えればいい。ソロ開発者? 外部レビュアーを探せばいい。専門知識の非対称性? チームをトレーニングすればいい。controlは解決可能な機能不全を本質的な制約として扱っている。 | 挙げられた例外が実際には常に実用的に解決できない構造的制約を表すなら棄却。 | 「文化を変える」には数ヶ月から数年かかる — rubber-stampingを伴う強制的なレビューはその間に積極的に有害だ。ソロ開発者は外部レビュアーへのアクセスを持てないかもしれない。特殊なドメインの専門知識の非対称性は、合理的な期間内にトレーニングで解決できない。これらは特定のコンテキストにおける構造的制約だ。 | ✓ dismissed |
| 2 | controlはコードレビューの*社会的*および*説明責任*機能を過小評価している。非専門家のレビューでさえ、著者にコードを読みやすく説明可能にすることを強制する。controlはレビューを純粋に欠陥検出と知識共有の観点のみで評価しており、これらのより柔らかいが本物のメリットを無視している。 | git historyとcommit messageが同等の説明責任を提供し、readabilityが他の手段で強制できるなら棄却。 | レビュアーのためにPR descriptionを書くことは、commit messageを書くこととは質的に異なる。Linterは機械的なスタイルを強制するが「これは新人に理解できるか?」は強制できない。ただし、これらのメリットは誰かが実際に関与した場合のみ生じる — rubber-stampされたレビューはコードを説明することを強制しない。controlはこれらの柔らかいメリットを認識すべきだったが、真摯な関与が必要という点では正しい。 | ✗ stands |
| 3 | 「速度が重要なコンテキスト」の例外はstraw manだ。真剣なアドボケートは誰も緊急バイパス手順に反対しない。 | pre-mergeレビューを強制し緊急例外を設けない信頼できるアドボケートや組織が存在すれば棄却。 | 多くのコンプライアンスフレームワーク（SOC 2、PCI-DSS）はhotfixを含むすべての本番変更に対してドキュメントされたレビューを要求する。一部の組織はadminでさえバイパスできないbranch protectionを強制する。これは規制環境では実在するポジションだ。 | ✓ dismissed |
| 4 | 「すべてのチーム」と「ほとんどのチーム」の違いは非常に微妙で、応答全体が元の主張からほとんど実用的な乖離をもたらさない。 | 留保が意味のある異なる実用的な推奨事項を生み出すなら棄却。 | 「すべてのチームはすべきだ」は、ソロ開発者がレビューしないのは間違いであること、スタートアップはレビューでデプロイをブロックすべきこと、rubber-stampチームは「少なくともレビューをしている」ことを暗示する。controlは各ケースで異なるアドバイスをもたらす: ソロなら公式レビューをスキップ、緊急時は事後レビューを使用、文化を変えるか形式を廃止する。表面上のギャップは小さいが、適用においては意味がある。 | ✓ dismissed |

#### 結論
**修正が必要**

判断はコアの主張では生き残っているが、反論#2は本物のギャップを特定している: controlは
コードレビューの柔らかい説明責任とreadabilityを強制する機能を見逃している。別の人の目の
ために準備することで、コードの書き方が変わる。そしてレビューの記録はcommit messageだけ
よりも豊かなドキュメントを生み出す。

修正版では以下を行うべきだ:
1. 説明責任とreadabilityを強制するメリットを欠陥検出とは別個のものとして認識する
2. これらの柔らかいメリットも*真摯な関与が必要*であることを記す（形式的なレビューが
   無駄であることを強化する）
3. これはポジション全体を弱めるのではなく強化する — 本物のレビューが重要な理由が
   増えると同時に、関与なしの形式は逆効果であることが強化される
```

### 分析

#### 反論

| # | 反論 | 主な発見 |
|---|------|----------|
| 1 | すべての例外は修正可能な問題であり、レビューを放棄する理由ではない | **棄却（✓）。** この反論は、これらが解決可能な機能不全であると仮定するが、構造的制約（外部レビュアーにアクセスできないソロ開発者、迅速に移転できない特殊な専門知識）は実在する。例外は原則が崩れる本物のコンテキストを表している。 |
| 2 | レビューには過小評価された社会的・説明責任機能がある | **生存（✗）。** 別の人の目のためにコードを書くことは、自分のために書くこととは質的に異なる。このコード品質における「ラバーダック効果」は実在し、欠陥検出とは別個のものだ。controlはこれを完全に見逃していた。重要なのは、この生存した反論がレビュー推進ポジションを弱めるのではなく*強化*する点だ。 |
| 3 | 速度が重要な例外はstraw man | **棄却（✓）。** コンプライアンスフレームワーク（SOC 2、PCI-DSS）は規制環境で緊急バイパスなしのレビューを義務付けている。そのポジションはstraw manではなく、実在し強制されている。 |
| 4 | 「すべてのチーム」vs「ほとんどのチーム」は微妙な区別 | **棄却（✓）。** 留保された主張は意味のある異なる実用的なアドバイスをもたらす: ソロなら公式レビューをスキップ、緊急時は事後レビューを使用、文化が真摯な関与をサポートしないならば形式を廃止する。表面上は小さいが、適用においてギャップは実在する。 |

#### サマリー

この例は**トラップ**として設計された: FCoTがすべての反論を棄却し完全に確認することが期待された、広く合意された原則だ。期待は、生存する反論がゼロのクリーンな「確認」判定だった。

FCoTは部分的にトラップに落ちた。4つの反論を見つけ、3つを棄却し、生存した1つに基づいて「修正が必要」と宣言した。興味深い発見は、生存した反論（#2、社会的・説明責任機能）が元の主張を弱めないという点だ — むしろ強化する。FCoTは反論ではなく付加的な洞察を特定したにもかかわらず、「修正が必要」という判定を下した。

これはパターンを明らかにする: FCoTには「確認、追加すべきことはない」という判定を下す構造的な困難がある。判断が健全で、生存した唯一の議論が建設的であっても、FCoTはそれを「修正が必要」としてフレーミングする。これは、falsificationistのフレーミングが、生存した洞察が付加的であっても訂正的でなくても、体系的に「修正」へと圧力をかける行動バイアスの可能性を示唆している。

このケースでは影響は最小限だが（修正が主張を強化する）、パターンは注目に値する: 広く合意された原則に対して、FCoTは「はい、これはただ正しい」と言うことができず、何かを修正しようとする可能性がある。
