# TypeScript `any`

> **題材:** 「TypeScript で `any` は絶対に使ってはいけない」

## FCoT なし

```
> TypeScript で `any` は絶対に使ってはいけない。

この考え方の趣旨にはおおむね賛成だが、「絶対に」は言いすぎだ。正確に言うとこうなる:

`any` が危険な理由: `any` は TypeScript の型システムを完全にバイパスしてしまう。一度コードに
`any` が入り込むと、それは伝播していく — 触れたすべての値をチェックなしの型で汚染してしまう。
そもそも TypeScript を使う意義を損なうことになる。

`unknown` のほうがほぼ常に適切: 型が本当に分からない場合、`unknown` を使えば利用前に型を
絞り込むことが強制される。「これが何か、まだ分からない」という状態を型安全に表現できる型だ。

ただし `any` に正当な用途があるケースはある:
- JavaScript からの移行: すべてを一気に型付けするのは現実的でない。
- 型定義のないサードパーティライブラリ。
- TypeScript の型システムでは表現しきれない複雑な型パズル。
- 型付けで速度が落ちる場面でのラピッドプロトタイピング。

本来の原則: デフォルトでは `any` を避ける。どうしても使う場合は、怠慢からではなく
意識的な判断として使うこと。後で見直す技術的負債として扱うべきだ。ESLint の
`@typescript-eslint/no-explicit-any` ルールで強制し、`any` が本当に正当化される箇所には
`eslint-disable` コメントをピンポイントで記述する。
```

## FCoT あり

```
> /fcot

**判断は健全。**

5 つの反論すべて棄却。「`any` は絶対に使うな」は言いすぎだ。本来の原則 —
`unknown` を優先し、`any` は意識的に、定義された境界の中で使う — は
反証に耐えた。
```

### 考察

#### 反論のテーブル

| # | 反論 | 棄却条件 | 検証 | 結果 |
| --- | --- | --- | --- | --- |
| 1 | `unknown` は `any` として挙げた正当なユースケースをすべてカバーできる。移行、動的データ、サードパーティライブラリ — いずれも `unknown` で対応できる。 | `any` が本当に必要で、`unknown` では不合理な複雑さなしに代替できない実践的シナリオが少なくとも 1 つ存在すれば棄却。 | レガシー JS の内部実装とのインターフェース、特定のメタプログラミングパターン、または `as unknown as T` が表面的にしか安全でない場合 — 本当の規律はキャストではなく周囲の型契約にある。実際のギャップが存在する。 | ✓ |
| 2 | 「プロトタイピング」を正当なケースとして挙げることは無頓着な使用を是認してしまう — 「一時的な」`any` は結局クリーンアップされない。 | 回答がプロトタイピングでの使用を後で見直すべき技術的負債として扱っているなら棄却。 | 回答は「後で見直す技術的負債として扱う」と明示している。`any` をそのまま残すことを推奨していない。 | ✓ |
| 3 | 「絶対に」は論理的な絶対値ではなく実践的なヒューリスティックだ。技術的に不正確でも、より厳しいルールのほうが良い結果をもたらす。 | 回答が `any` への強い忌避を保ちつつ、ルールを実質的に弱めていないなら棄却。 | 回答は `any` を逃げ道として位置付け、`unknown` を優先の代替として挙げ、`any` を技術的負債と呼んでいる。是認ではなく調整だ。 | ✓ |
| 4 | 型のないサードパーティは弱い理由だ — `@types/*` パッケージは大半のライブラリに存在し、ローカルの `.d.ts` スタブのほうがベターなプラクティスだ。 | 型のないサードパーティが現代の TypeScript において実際に起こる非自明なシナリオであれば棄却。 | ニッチ・レガシーなライブラリ、内製ツール、自動生成 SDK クライアントは不完全な型のまま存在する。多少誇張気味だが主張は成立する。 | ✓ |
| 5 | 回答は `any` と型アサーション (`as any`) を混同している。変数型の `any` は安全でなく伝播するが、ターゲットを絞ったキャストの `as any` はローカルに留まる。 | この区別がコアの判断を覆さなければ棄却。 | 区別は実在し注記する価値があるが、コアの判断（「意識的 vs. 無頓着」）は残る。精度のギャップであり、反証ではない。 | ✓ |

#### まとめ

これは **FCoT が初期判断を確認する** 例だ。AI のポジション — 「デフォルトで `any` を避け、意識的に使う」 — はもともと十分に根拠があった。5 つの反論はすべて棄却された。

それでも FCoT は価値を加えた: 精度のギャップ（変数型の `any` とターゲットを絞った `as any` では伝播の挙動が異なる）が特定され、アドバイスをより精確にできる可能性が示された。ただしコアの判断は反証に耐えて無傷だった。
